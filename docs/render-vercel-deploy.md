# Deploying NADAA — Render (backend) + Vercel (frontend)

This is the concrete, click-by-click guide for the production topology:

- **Backend** — the 18 Go microservices run as **Docker web services on Render**, declared in [`render.yaml`](../render.yaml) (a Render Blueprint).
- **Frontend** — the 6 Vite web apps run on **Vercel**, one project per app, each with a [`vercel.json`](../apps/citizen-web/vercel.json).

The services use in-memory stores (stdlib-only Go), so **no database is required** for the MVP — but data resets on every redeploy. Delivery/payment providers stay in **sandbox** until you supply Arkesel/Paystack keys.

---

## 1. Deploy the backend on Render

### 1.1 Create the Blueprint

1. Push this repo to GitHub (already done) and connect it in Render.
2. In Render: **New → Blueprint**, pick this repo. Render reads `render.yaml` and lists all 18 services.
3. On first apply, Render prompts for every `sync: false` value (secrets + inter-service URLs). You can leave the inter-service URLs blank for now (services boot without them — those calls are optional enrichment) and fill them in step 1.3.
4. **Apply**. Each service builds from its `Dockerfile` (context = repo root) and deploys.

Notes baked into the blueprint:

- **Region** is `frankfurt` (closest Render region to Ghana). Change it in `render.yaml` if you prefer.
- **Plan** is `starter` (~$7/service/month). The free plan spins services down after 15 min idle — impractical for 18 interdependent services, so `starter` is the default. Change `plan:` if you want.
- **Port** — every service binds Render's injected `$PORT` (via `resolveListenAddr`); no per-service port config needed.
- **Health checks** — `/healthz` for most; `/health` for `route` and `damage-claim` (already set per service).
- **Auth token secret** — `NADAA_AUTH_TOKEN_SECRET` is auto-generated by Render (`generateValue`). Mock-actor headers and dev-OTP exposure are forced off.

### 1.2 Set the provider secrets (when your keys arrive)

In the Render dashboard, set these on the relevant service, then it flips from sandbox to live automatically (see [notification-service](../services/notification-service/README.md) / [donation-service](../services/donation-service/README.md) READMEs):

| Service | Env var | To go live |
| --- | --- | --- |
| notification | `NADAA_ARKESEL_API_KEY` | also add env `NADAA_SMS_PROVIDER=arkesel` |
| notification | `NADAA_EXPO_ACCESS_TOKEN` | optional; add `NADAA_PUSH_PROVIDER=expo` |
| donation | `NADAA_PAYSTACK_SECRET_KEY` | also add `NADAA_PAYMENT_PROVIDER=paystack` and `NADAA_PAYSTACK_CALLBACK_URL` |

### 1.3 Wire the inter-service URLs (optional enrichment)

A few services call siblings. After the first deploy, copy each target's URL from Render (**Connect → Internal** for private-network calls, e.g. `http://nadaa-ml-service:10000`, or the public `https://…onrender.com`) and set the matching env var. Keep the **same path shape** as the local default shown below.

| Service | Env var | Points at | Value shape |
| --- | --- | --- | --- |
| risk | `NADAA_ML_API_URL` | ml | `http://<ml>/api/v1` |
| notification | `NADAA_ALERT_SERVICE_URL` | alert | `http://<alert>/api/v1` |
| notification | `NADAA_INCIDENT_SERVICE_URL` | incident | `http://<incident>/api/v1` |
| damage-claim | `INCIDENT_SERVICE_URL` | incident | `http://<incident>/api/v1` |

## 2. Deploy the frontends on Vercel

Each app is a **separate Vercel project** pointing at the same repo, distinguished by **Root Directory**. Each app already has a `vercel.json` (SPA rewrites, asset caching, security headers, pnpm-workspace install).

For each of `marketing-web`, `citizen-web`, `authority-dashboard`, `dispatcher-web`, `admin-web`, `agency-web`:

1. Vercel: **Add New → Project**, import this repo.
2. Set **Root Directory** to `apps/<app>` (e.g. `apps/citizen-web`). Vercel detects the pnpm workspace and installs from the repo root automatically.
3. Framework preset **Vite** is auto-detected (output `dist`). The `vercel.json` handles the rest.
4. Add the app's **Environment Variables** (section 2.1) — these are baked into the bundle **at build time**, so set them before deploying and redeploy after any change.
5. **Deploy**. Note the resulting `https://<project>.vercel.app` domain.

## 3. Close the loop (CORS)

The frontends call the APIs from the browser, so the backend must allow their origins.

1. After the Vercel projects are live, collect their domains.
2. In Render, edit the **`nadaa-web-origins`** environment group → set `NADAA_ALLOWED_ORIGINS` to the comma-separated Vercel domains, e.g.
   `https://nadaa-citizen.vercel.app,https://nadaa-authority.vercel.app,https://nadaa-dispatcher.vercel.app,https://nadaa-admin.vercel.app,https://nadaa-agency.vercel.app`
3. Render redeploys the services using the group. Done — the browsers can now call the APIs.

Until you set this, `NADAA_ALLOWED_ORIGINS` is `*` (open CORS) so things work during bring-up; **lock it to your domains before going public.**

---

## Deploy order (resolves the frontend↔backend dependency)

1. Render Blueprint → get the backend `.onrender.com` URLs.
2. Set those URLs as the Vercel `VITE_*` vars → deploy the frontends → get the `.vercel.app` domains.
3. Set those domains as `NADAA_ALLOWED_ORIGINS` in the Render `nadaa-web-origins` group.
4. Set provider secrets (Arkesel/Paystack) whenever they're ready.

## Caveats (MVP)

- **Data is in-memory** — every service resets its data on redeploy/restart. Wire a real datastore before production.
- **Imagery uploads are ephemeral** — `imagery-service` writes to the container filesystem (`/app/uploads`), lost on redeploy. Attach a Render disk mounted at that path for durability.
- **Providers default to sandbox** — no real SMS/push/payments until keys are set (section 1.2).
